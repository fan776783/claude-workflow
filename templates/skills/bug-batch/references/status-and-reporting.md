# 状态流转与汇总模板

当 `FixUnit` 完成修复后，`bug-batch` 需要先推进到 `处理中`，在人工确认后再推进到 `待验证`。此文件沉淀状态流转示例和最终汇总模板。

## 1. 流转到处理中

若 MCP 支持批量更新，可参考：

```text
batch_update_issue_states(
  issue_numbers: ["p003", "p001", "p002"],
  target_state: "处理中",
  comment: "FixUnit FU-001 已完成代码修复与验证，等待人工确认后流转待验证"
)
```

若只支持单条更新，可顺序执行等价调用：

```text
update_issue_state(
  issue_number: "p003",
  target_state: "处理中",
  comment: "FixUnit FU-001 已完成代码修复与验证，等待人工确认"
)
```

## 2. 人工确认卡点

```markdown
## FixUnit 已修复，等待人工确认

### 修复单元
- FixUnit: FU-001
- 主缺陷: p003
- 直接覆盖缺陷: p003, p001
- 重复缺陷: p002

### 修复摘要
- 根因: token 刷新链路失效
- 修改文件: auth.ts, session-store.ts
- 剩余风险: 低

### 建议人工验证
- 验证主流程登录刷新
- 验证历史重复问题 p002 是否一并消失
- 验证依赖单元 FU-002 的前置条件是否已满足

## 人工是否确认该 FixUnit 可以进入待验证？(Y/N)
```

## 3. 流转到待验证

```text
batch_update_issue_states(
  issue_numbers: ["p003", "p001", "p002"],
  target_state: "待验证",
  comment: "FixUnit FU-001 已经人工确认，可进入待验证"
)
```

## 4. 批量汇总报告

```markdown
## 批量修复报告

### 修复单元统计
- 总单元数: N
- 成功完成: X
- 人工介入: Y
- 阻塞未执行: Z

### FixUnit 视图
| FixUnit | 主缺陷 | 覆盖缺陷 | 状态 | 修改文件 | 根因摘要 |
|---------|--------|----------|------|----------|----------|
| FU-001 | p003 | p003,p001,p002 | 已流转待验证 | auth.ts | token 刷新链路失效 |
| FU-002 | p004 | p004 | 人工介入 | - | 依赖验证失败 |

### Issue 视图
| 工单号 | 所属单元 | 角色 | 最终状态 | 说明 |
|--------|----------|------|----------|------|
| p003 | FU-001 | 主缺陷 | 待验证 | 直接修复 |
| p001 | FU-001 | 关联缺陷 | 待验证 | 共享根因一并覆盖 |
| p002 | FU-001 | 重复缺陷 | 待验证 | 继承主修复结果 |

### 失败 / 阻塞项
- FU-002 / p004: 依赖前置接口恢复后再验证
```
