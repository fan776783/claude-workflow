# Spec: 执行流程精简化删可恢复

> 统一规范文档：需求范围 + 设计决策 + 验收标准

## 1. Context

- 原始需求来源: `.claude/temp/lean-execute-prd-0527.md`（grill 对齐产物）
- 本次改动对象: claude-workflow repo 自身（框架层重构），非业务项目
- 依赖证据: reelmatefrontend journal 显示实际 run 均为 5–7 task 单会话跑完（session-014=7、session-015=5），跨会话 resume 实证极少

### 1.1 Problem Statement

workflow-execute 当前是"可恢复治理型状态机"：per-task ~7 次 CLI 往返中约 4 次（governor `decide`/`decide-post-execution`、`quality_review pass`、`advance` 的 state 写）纯为 resume + 跨会话审计存在；state 文件 80%（实测 11.8KB）是 `quality_gates` per-task 审计记录。resume 既然罕见，这层 durable + 机器化 gate 是几乎不回本的固定成本。对齐 superpowers `subagent-driven-development`（单会话、controller 持全 plan、末尾全量终审），把 execute 收敛为单一 lean 路径。

### 1.2 Assumptions

- 用户接受跨会话 resume 退化为手动（开新会话 + 读 plan + git log 定位进度）
- per-task review 审计链可由"末尾全量终审 + git history"替代
- 删除边界限于 execute 阶段 + `/workflow-review`；spec/plan/delta 阶段的 state 字段不动（属各自 skill 主权，见 §2.2）

## 2. Scope

### 2.1 In Scope

- R-002: 删除 per-task quality gate 持久化——execute 不再调 `quality_review.js pass/fail` 写 `state.quality_gates`；reviewer PASS 仅内存确认即继续

## 4. User-facing Behavior

### 4.1 Primary Flow

`/workflow-execute` lean 主路径：
1. controller 读极简 state + 一次性读 plan 持全 task 切片
2. 逐 task（顺序）：构 implementer prompt（`task_text` 切片 + 策展 context + soft write-scope prose）→ 派 fresh implementer subagent → 派 reviewer subagent（AC→质量两 phase）→ `verification.js` 验证 → 内存确认 PASS → `advance`（更新 plan checkbox + `progress.completed`）
3. 所有 task 完成 → inline 派 final reviewer subagent（整 branch diff vs spec）→ 通过 → `status=completed`
4. 无 governor CLI 往返；context 压力由 controller 内联 banner 提示

### 4.2 Error and Edge Flows

- reviewer 终态 FAIL（implementer↔reviewer loop 上限 3）→ `halted` + `halt_reason='review-loop'`，展示 issues 等用户介入
- final reviewer 发现跨 task 集成问题 → 不自动回退；展示 + 用户决策（另起修复 / accept）
- implementer 越界改文件 → 自报 `DONE_WITH_CONCERNS` → reviewer 复核裁定
- context 满 → controller banner 提示开新会话；新会话靠 plan + git log + `progress.completed` 手动定位（无自动 resume）

### 4.3 Observable Outcomes

- 每 task 模型面 CLI 往返 ~7 → ~2
- state 文件 ~22KB → ~3KB（`quality_gates` 蒸发）
- 无 `review_pending` 中间态；execute 跑完即 `completed`
- `/workflow-review` 命令消失；branch 级单审走 `/diff-review`

### 5.2 Data Models

极简 state schema：
- **保留**：`project_id / status / current_tasks / plan_file / spec_file / progress{completed,skipped,failed,blocked} / task_runtime / unblocked / sessions / git_status / halt_reason / failure_reason / consecutive_count / deviation_log / created_at / updated_at`
- **删除**：`quality_gates`、`continuation`、`review_report_path`（`contextMetrics` 随 governor budget 一并退役）
- **保留但本次不动**（spec/plan/delta 拥有）：`review_status / context_injection / delta_tracking / contract_digest_path / discussion / ux_design / api_context`
- **status 枚举**：`idle → spec_review → planned → running → completed`（删 `review_pending`）；`halted` 仍存（halt_reason: `failure`/`dependency`/`review-loop`；删 `governance`）

### 5.3 Technology Choices

- 无新依赖；纯 Node.js CLI + Markdown skill 改造
- 架构对齐 superpowers `subagent-driven-development`：controller 一次性持全 plan、implementer 收 full text、末尾派 final reviewer subagent 跑整 branch

### 5.4 Risks and Trade-offs

- 删 governor 后无可靠 in-session token 预算 → context 压力判断退化为启发式/用户感知（见 §9.4）
- 删机器 write-scope → 大共享前端越界风险残留，靠 reviewer 兜（已接受，见 §2.1 R-007）
- 60 处 `workflow-review` 引用清理遗漏 → 悬挂链接；以 grep 全量归零为验收项（§7）
- glossary 改定义可能触发 `validate.js` 全量 drift 扫描连带其它 normative 文档
