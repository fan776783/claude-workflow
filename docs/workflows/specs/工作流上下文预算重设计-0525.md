# Spec: 工作流上下文预算重设计

> 统一规范文档：需求范围 + 设计决策 + 验收标准

## 1. Context

- 原始需求来源: grill 对齐描述（暂存 /tmp/wf-context-budget-req.md）
- 改动对象: agent-workflow 体系自身代码（`core/` 下 hook / CLI / skill），dogfood 类 infra 变更，无终端用户 UI。
- 内部依赖: `core/hooks/pre-execute-inject.js`（PreToolUse(Task) hook）、`core/utils/workflow/{task_runtime,workflow_cli,workflow_types,path_utils}.js`、四个 phase skill SKILL.md。
- 外部依赖: 无新增。

### 1.1 Problem Statement

workflow 全链路实测两处上下文浪费：① **execute 重复读**——每 task 起 fresh implementer subagent，各自从零读共享源码（store/组件/types），N task 重读 N 次（计费 token，~40-80K）；② **phase 间全文回灌 + Explore 报告灌主会话**——phase 间靠重读完整 spec/plan 定向，Explore 调研报告整篇进主会话（实测单份 16.9K 字符）。现有 `pre-execute-inject.js` 已能在 dispatch 边界把 task block + spec + code-specs digest 注入 subagent（内容绕过主会话），但**不注入共享源码 contract**，也无 phase 间压缩机制。

### 1.2 Assumptions

- contract-digest 整份注入，不做 per-task 切片（contract 本就跨 task 共享）。
- handoff 存 `~/.claude/workflows/{pid}/`，与 plan/state/reports 同级，不入项目 git。
- contract digest 由 spec 阶段产出，plan 阶段可追加。

## 2. Scope

### 2.1 In Scope

- R-001: **contract-digest 落盘**。workflow-spec 代码分析的 contract 产出写入 `~/.claude/workflows/{pid}/contract-digest.md`（受影响文件 + 关键签名 + 共享 store/type shape，bounded ~3000 字符），不再整篇进主会话。
- R-002: **state 字段 + CLI**。`workflow-state.json` 增 `contract_digest_path`（`string|null`，默认 `null`）；新增 CLI `set-contract-digest-path --path <p> [--unset]`，复刻 `review_report_path` / `cmdSetReportPath` 模式。
- R-003: **hook 注入 contract 块**。`task_runtime.js` 加 `getContractDigest(runtime)`；`pre-execute-inject.js` `buildTaskContext` 对 `kind ∈ {implement, check}` 新增 `<task-contract>` 块注入该 digest，**augment** 现有 `<spec-context>` / `<project-code-specs>`，不替换。内容走 hook→subagent，绕过主会话。
- R-004: **phase handoff 落盘**。spec→plan / plan→execute / execute→review 三边界各写 ≤20 行 handoff（decisions / rejected / risks / contract-digest 指针）到 `~/.claude/workflows/{pid}/handoff/{from-phase}.md`。
- R-005: **handoff 消费**。下一 phase 的上下文加载步**先读 handoff 定向**，全文 spec/plan **按需再读**；配合各边界已有 `/clear` 提示 → fresh resume 无损。
- R-006: **四个 phase skill 协议更新**。`workflow-spec` / `workflow-plan` / `workflow-execute` / `workflow-review` SKILL.md 增加产出/消费 handoff + 落盘 contract digest 的步骤约定。
- R-007: **回归保护**。现有 hook 的治理 gate（spec-review gate / status 判定）、role 分类（research/check/implement）、既有注入块（task block / spec / code-specs）行为不变。

- contract-digest per-task 切片 —— 默认整份注入。

## 3. Constraints

> 不可协商的硬约束，来自原始需求或技术限制。

- C-1: `contract_digest_path` 必须复刻 `review_report_path` 既有模式（同 `string|null` 顶层字段、`workflow_types.js` normalize 默认、`writeState` 持久化、`--path`/`--unset` CLI flag 风格），不引入新的 state 序列化方式。
- C-2: contract digest bounded ~3000 字符，**由 `getContractDigest` 在读时强制截断**（不依赖写入方），沿用 `getSpecContent` / `renderSpecFiles` 既有读时 maxChars 惯例。
- C-3: contract-digest 整份注入，**不做 per-task 切片**。
- C-4: handoff 落 `~/.claude/workflows/{pid}/handoff/{from-phase}.md`，**不入 state schema**（命名约定发现，覆盖式写入）。**每个 handoff 顶部带 freshness header**：`from` / `to` / `state_updated_at` / `spec_file` / `plan_file`；消费方比对当前 state，不符则忽略 handoff 改读全文（解 stale 失败模式）。
- C-5: contract 与 handoff 内容均走 hook / disk → subagent，**不经主会话粘贴**；语义上 contract（既有代码复用面）区别于 spec（需求）与 code-specs（规范），三者并存不重复。
- C-6: 不破坏 ADR 0002 的 fresh-subagent-per-task 主路径与"只读 fan-out only"约束（见下）。
- C-7: `<task-contract>` 正文注入前必须 sanitize（转义 `</task-contract>` 及 `<system…>`/`<workflow-context…>` 等标记），复用/扩展 `task_runtime.js` 既有 `sanitizeCodeSpecsBody`，与 `<project-code-specs>` 注入对等，防内容破坏块边界。

## 4. User-facing Behavior

> 「user」= 运行 workflow 的开发者 / 模型；行为以 workflow 运行时可观测现象描述。

### 4.1 Primary Flow

- spec 阶段：代码分析产出 contract → 落盘 `contract-digest.md`，state.`contract_digest_path` 被设置；主会话只见落盘确认 + 短摘要，不见全文。
- execute 阶段：派发 implementer / reviewer 时，subagent prompt 自动含 `<task-contract>` 块（hook 从落盘 digest 注入），fresh subagent 无需重读共享源码。
- phase 切换：每个边界（spec→plan / plan→execute / execute→review）落 `handoff/{from-phase}.md`；下一 phase 启动时先读 handoff 定向，再按需读全文 spec/plan。`/clear` 后 resume 信息无损。

### 4.2 Error and Edge Flows

- contract-digest 文件缺失 / 路径未设置 → hook 跳过 `<task-contract>` 注入（不阻断派发），行为退化为现状。
- digest 超 bound → `getContractDigest` 在**读时**按 maxChars 截断（不依赖写入方自觉），与 `getSpecContent` / `renderSpecFiles` 既有读时截断一致（codex P1 #1）。
- digest 正文含 `</task-contract>` / `<system…>` / `<workflow-context…>` 等标记 → 注入前经 sanitize 转义，防止破坏块边界（codex P1 #2，对齐 `sanitizeCodeSpecsBody`）。
- handoff 文件缺失 → 下一 phase 回退读全文 spec/plan（现状行为），不报错。
- handoff **过期**（文件在但 header 的 `state_updated_at` / `spec_file` / `plan_file` 与当前 state 不符）→ 消费方忽略 handoff，回退读全文（codex P1 #3，防"自信地错")。

### 4.3 Observable Outcomes

- execute 派发的 subagent prompt 含 `<task-contract>` 块且内容来自落盘 digest（非主会话粘贴）。
- Explore 报告不再整篇出现在主会话；主会话只持有落盘指针 + 短摘要。
- 三个 phase 边界各有一个 handoff 文件；仅凭 handoff 可定向恢复。
- 现有治理 gate / role 分类 / 既有注入块行为无回归。

## 5. Architecture and Module Design

- 以现有代码结构为基线，采用最小必要改动完成需求
- 优先复用现有模块、状态流转与验证能力

### 5.1 Module Responsibilities

| Module | 职责 | 落点（dev 仓库 `core/`） |
|--------|------|------------------------|
| ContractDigestWriter | spec 代码分析后把 contract 写 `contract-digest.md` + 设 state 路径 | `workflow-spec/SKILL.md` 协议 + CLI `set-contract-digest-path` |
| ContractDigestStore | state 字段 `contract_digest_path` + `cmdSetContractDigestPath` + normalize 默认 | `utils/workflow/workflow_cli.js`、`workflow_types.js` |
| ContractDigestInjector | `getContractDigest(runtime)`（读时 maxChars 截断 + sanitize body）+ `<task-contract>` 注入块 | `utils/workflow/task_runtime.js`、`hooks/pre-execute-inject.js` |
| HandoffStore | `handoff/{from-phase}.md` 读写（plain file I/O + 目录约定） | `utils/workflow/path_utils.js` helper + skill 协议 |
| PhaseProtocol | 四个 phase 的产出/消费 handoff 步骤 | `workflow-{spec,plan,execute,review}/SKILL.md` |

### 5.2 Data Models

- `state.contract_digest_path`: `string | null`（默认 `null`），绝对或相对 workflow 目录路径。
- `contract-digest.md`（落 `~/.claude/workflows/{pid}/`）: markdown，章节 = 受影响文件清单 / 关键签名 / 共享 store·type shape；bounded ~3000 字符。
- `handoff/{from-phase}.md`（`{from-phase}` ∈ spec/plan/execute）: 顶部 freshness header（`from` / `to` / `state_updated_at` / `spec_file` / `plan_file`）+ markdown ≤20 行正文（`## Decisions` / `## Rejected` / `## Risks` / contract-digest 指针）。
- `<task-contract>` 注入块: `<task-contract>\n{sanitize(getContractDigest 读时截断后的正文)}\n</task-contract>`，在 `buildTaskContext` 中排在 `<spec-context>` / `<project-code-specs>` 同级。

### 5.3 Technology Choices

- 纯 Node.js，复用现有 `readFile` / `path_utils` helper，**无新依赖**。
- CLI 子命令复刻 `cmdSetReportPath` / `set-report-path`（`optionOrArg(--path)` + `--unset` + `writeState`）。
- hook 注入复用现有 `buildAllowResult(patchedToolInput)` 的 `tool_input.description` patch 机制，不新增输出通道。

### 5.4 Risks and Trade-offs

- contract 与 spec/code-specs 内容**重叠** → 靠语义边界（contract=既有代码复用面）+ 各自 XML 标签区分，review 时人工校验不重复。
- handoff **漂移**（写了不读 / 读到旧版）→ 覆盖式写入 + 固定命名降风险；消费方按需回退读全文兜底（不强依赖 handoff 新鲜度）。
- 注入使 subagent prompt 变长 → bounded maxChars 控制；整份注入接受跨 task 冗余（contract 本就共享，per-task 切片的复杂度不划算）。

## 9. Open Questions & Dependencies

### 9.1 需求澄清记录

> 来源：grill 对齐（4 问）+ 默认假设确认。

| 维度 | 问题 | 答案 | 影响 |
|------|------|------|------|
| 问题域 | 解决哪个上下文浪费 | 两者合一：execute per-task contract 注入 + phase handoff digest | 范围=两套机制 |
| digest 来源 | contract digest 谁产出、怎么 curate | 复用 Explore 产出，落盘自动注入（非手动 curation） | 顺带修 Explore 灌主会话 |
| handoff 范围 | 覆盖哪些边界、装什么 | 3 边界 + 方向性 digest（decisions/rejected/risks）+ 全文按需读 | 写/读点落各 phase 终末/启动步 |
| 注入粒度 | digest 整份还是 per-task 切 | 整份注入，不切片 | 简化，接受跨 task 冗余 |
| handoff 存储 | 项目内还是 workflow 目录 | `~/.claude/workflows/{pid}/`，不入 state schema | 命名约定发现 |

### 9.2 方案选择

**选定方案**：保持 fresh-subagent-per-task 隔离不变，用 hook 在 dispatch 边界注入落盘 contract digest（杀重复读）+ phase handoff digest（杀全文回灌）。

**被排除方案**：

| 方案 | 排除原因 |
|------|---------|
| 单 implementer 干完所有 task（B1） | context 单调累积 → 长上下文质量退化 + compaction 风险 + 丢隔离；省的重复读 < 亏的隔离 |
| 砍 per-task review，末尾批量 review（B2） | 错误发现晚 → revise 级联回炒多 task，比省下的 reviewer spin-up 更贵 |
| dispatch-merge（按耦合合并执行）| revise blast radius 变大 + 丢 cluster 内隔离 + 对主上下文几乎无效；列为独立后续评估 |
| Trellis 式手动 `{file,reason}` curation | 加人工负担；auto-derive 已能复用 Explore 产出 |
| OMC deslop 质量门 | 质量维度，与上下文主题正交，本次出范围 |

### 9.3 未解决依赖

无未解决外部依赖——纯内部 infra 改动，不依赖后端接口 / 第三方就绪。

### 9.4 Open Questions

- 若 execute 中途发现 contract digest 不全（spec/plan 漏了某共享面），是否需要 execute 侧的 digest 刷新路径？当前定为 spec 产出 + plan 可追加，execute 不写。倾向：execute 发现缺口走现状（implementer 自行补读），不引入 execute 写 digest，避免 contract 来源多头。留待 plan 阶段确认。
