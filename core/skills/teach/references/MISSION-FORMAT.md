# MISSION.md 格式

`MISSION.md` 放 workspace 根目录,记录用户学这个主题的 _原因_。每个教学决策——下一步教什么、引哪些资源、设计哪些练习——都应回溯到这份文档。

## 模板

```md
# Mission: {Topic}

## Why
{1-3 句。用户追逐的具体现实目标。拿到这项技能后,生活或工作里什么会变?避免"to understand X"这类抽象表述——追问底层 outcome。}

## Success looks like
- {一件具体、可观察的、用户将能做到的事}
- {另一件}
- {…}

## Constraints
- {时间、预算、已有承诺、学习偏好——任何限定方法的因素}

## Out of scope
- {用户明确现在不追的相邻主题——保护 zone of proximal development}
```

## Rules

- **一个 workspace 一个 mission。** 想学两个不相关的东西 = 两个 workspace。
- **具体优于抽象。** "十月跑完半马" 优于 "变得更健康";"给团队 ship 一个 Rust CLI" 优于 "学 Rust"。
- **对模糊回击。** 用户说不清 why → 先访谈再落笔。坏 mission 比没有 mission 更糟。
- **现实变了就修订。** mission 会变。用户目标移动时更新本文件,别让过期 mission 误导后续 session。
- **保持短。** `MISSION.md` 超过一屏 → 它已经从指南针变成了计划。
