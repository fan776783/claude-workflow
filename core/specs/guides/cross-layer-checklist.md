# 跨层检查清单

## 触发条件

在以下场景中必须执行此检查：

- 实现**跨越 3 个以上层**的功能（如 UI → API → Service → DB）
- 数据在层间传递时**格式发生变化**（如 DTO → Entity → ViewModel）
- 同一数据被**多个消费者**使用
- 修改**共享接口**（类型定义、API contract、数据库 schema）

## 检查清单

### 1. 数据流完整性

- [ ] 数据是否正确地流过**所有层**？（绘制简单数据流图）
- [ ] 每一层的数据**转换/映射**是否正确？
- [ ] nullable/optional 字段在每一层是否一致处理？
- [ ] 集合类型（数组/列表）是否在所有层保持正确的类型？

### 2. 错误传播

- [ ] 底层错误是否正确**向上层传播**？
- [ ] 错误信息是否在适当的层被**转换为用户友好的消息**？
- [ ] 是否存在被**静默吞掉的异常**？
- [ ] 超时/重试策略是否在正确的层实现？

### 3. 接口contract

- [ ] 层间接口的**类型签名**是否完整？
- [ ] 接口delta是否**向后兼容**？
- [ ] 是否有**缺失的验证**（输入在哪一层验证？输出在哪一层规范化？）

### 4. 副作用与状态

- [ ] 哪些层持有**可变状态**？状态delta是否安全？
- [ ] 是否存在**隐式的层间耦合**？（如通过全局变量、单例、事件总线）
- [ ] 事务边界是否明确？部分失败时的回滚策略？

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| 只修改一层的类型定义，忘记更新其他层 | 修改类型后搜索所有引用层 |
| 在 UI 层直接处理业务逻辑 | 保持层职责分离 |
| 忽略 null/undefined 在层间的传播 | 为每层定义明确的 null 策略 |
| 混淆内部数据模型和外部 API 模型 | 明确区分 Entity / DTO / ViewModel |
