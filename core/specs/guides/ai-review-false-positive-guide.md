# AI 审查误报指南

## 核心原则

> **"Most bugs come from 'didn't think of that', not from lack of skill."**
> 但同样地，大量 AI 审查发现也来自 **"没仔细看代码"**。

**预算：约 35% 的 AI 审查发现为误报。**

每个 CRITICAL 或 WARNING 级发现，在标记为"必须修复"前，必须对照源代码验证。

## 常见误报模式

### 1. 信任边界混淆 (Trust Boundary Confusion)

**表现**：AI 把内部数据当作外部输入，要求添加不必要的验证/消毒。

**识别方法**：
- 检查数据的**来源**：是用户输入？还是系统内部生成？
- 内部函数间传递的已验证数据**不需要重复验证**
- 检查上游是否已经做了验证

**正确处理**：
```
❌ "这个字段可能包含恶意输入" → 但它来自数据库的已验证数据
✅ 追踪数据到最初的输入点，确认验证发生在正确的边界
```

### 2. 忽略设计注释 (Ignoring Design Comments)

**表现**：AI 把文档或注释中标注的"有意为之"行为当作 bug。

**识别方法**：
- 检查相关代码是否有 `// intentional` / `// by design` / `// see: xxx` 注释
- 检查 PR 描述或 spec 文档中是否说明了此行为
- 检查 `.claude/specs/` 中是否定义了此约定

**正确处理**：
```
❌ "这个空 catch 块会吞掉异常" → 但注释写了 "// errors are non-fatal here"
✅ 阅读上下文注释，理解设计意图
```

### 3. 变量误读 (Variable Misreading)

**表现**：AI 没有追踪变量到实际定义，基于变量名猜测用途。

**识别方法**：
- AI 报告的变量类型是否与实际定义一致？
- 是否混淆了同名不同作用域的变量？
- 是否忽略了类型守卫或断言？

**正确处理**：
```
❌ "result 可能为 null" → 但上面有 if (result === null) return 的守卫
✅ 追踪变量到定义处，检查所有赋值和守卫
```

### 4. 过时上下文 (Stale Context)

**表现**：AI 基于之前版本的代码提出建议，忽略了最近的修改。

**识别方法**：
- 检查 `git diff` / `git log` 确认代码当前状态
- AI 引用的代码行号是否与当前文件匹配？
- 是否基于已被删除/修改的旧实现？

## 验证流程

对每个 CRITICAL / WARNING 发现执行：

```
1. LOCATE：定位 AI 指出的代码位置
2. TRACE：追踪相关变量/数据到源头
3. CONTEXT：阅读周围注释和相关文档
4. VERIFY：该发现在当前代码中是否成立？
5. DECIDE：
   - 成立 → 标记为有效发现
   - 不成立 → 标记为误报，记录原因
```

## 在 workflow 中的应用

### 质量关卡 Stage 2

在两阶段审查的 Stage 2（代码质量审查）中：
1. 对所有 CRITICAL 发现执行上述验证流程
2. 标记误报原因（属于哪种模式）
3. 仅将验证通过的发现计入最终结果

### `/workflow-execute --retry`

在调试失败任务时：
1. 检查失败原因是否来自 AI 审查
2. 如果是，先用此指南过滤误报
3. 仅针对真实问题执行修复
