# 验证清单生成系统 (v3.3.2+)

> Phase 0.6: 将结构化需求转换为可执行的验证清单

## 概述

验证清单生成系统在需求结构化提取（Phase 0.5）之后自动执行，将 9 维度的结构化需求转换为可执行的验收项，用于验证功能交付质量。

**与实现指南的关系**：
- **验收清单（Phase 0.6）**：用户视角的验收标准，关注"应该实现什么"
- **实现指南（Phase 0.7）**：开发者视角的实现路径，关注"如何测试和实现"

两者互补，共同指导开发和验收。

## 设计目标

1. **细节不丢失**：确保 PRD 中的所有验证细节（必填、格式、权限、交互等）都转换为验收项
2. **可执行性强**：每个验收项都包含明确的验证标准
3. **自动关联**：任务与验收项智能匹配，执行时自动引用
4. **结构化组织**：按场景、角色、功能模块分组，便于查找和验证
5. **职责分离**：验收清单关注验收标准，测试方法由实现指南提供

## 验证清单结构

### 1. 表单字段验证 (FormValidation)

**来源**：`RequirementAnalysis.formFields`

**转换规则**：
- 按场景（scene）分组
- 每个字段生成多个验证项：
  - 必填验证：空值时的提示信息
  - 格式验证：正则、类型、长度限制
  - 交互验证：placeholder、helper text、tooltip
  - 类型特定验证：文件上传、下拉选项、开关联动等

**输出格式**：
```markdown
### 1.1 创建弹窗

#### AC-F1.1 活动名称

**验证项**:
- [ ] 活动名称 为空时，显示提示: "活动名称为必填项"
- [ ] 活动名称 字符限制 50 字符
- [ ] 输入框显示 placeholder: "请输入活动名称"

**测试数据**:
| 输入 | 期望结果 |
|------|----------|
| （空值） | 显示错误提示: 活动名称为必填项 |
| 超过 50 字符的文本 | 禁止输入或显示错误提示 |
```

### 2. 角色权限验证 (PermissionValidation)

**来源**：`RequirementAnalysis.rolePermissions`

**转换规则**：
- 按角色分组
- 每个权限/限制生成验证项和测试步骤
- 包含数据范围验证（数据归属判定）

**输出格式**：
```markdown
### 2.1 管理员

#### AC-P1.1 管理员 - 删除活动

**验证项**:
- [ ] 管理员 可以执行 "删除活动" 操作
- [ ] 操作按钮/入口对 管理员 可见
- [ ] 执行操作后结果符合预期

**测试步骤**:
1. 使用 管理员 账号登录
2. 导航到相关功能页面
3. 验证 "删除活动" 操作可见且可执行
4. 执行操作并验证结果
```

### 3. 交互行为验证 (InteractionValidation)

**来源**：`RequirementAnalysis.interactions`

**转换规则**：
- 按交互类型（trigger）分组
- 每个交互生成验证项和前置条件
- 特殊处理：hover 延迟、弹窗层级、loading 状态等

**输出格式**：
```markdown
### 3.1 hover

#### AC-I1.1 删除按钮 - hover

**验证项**:
- [ ] hover 删除按钮 时，显示确认提示
- [ ] 延迟时间符合规格
- [ ] 显示提示信息: "确认删除该活动？"

**前置条件**: 用户有删除权限
```

### 4. 业务规则验证 (BusinessRuleValidation)

**来源**：`RequirementAnalysis.businessRules`

**转换规则**：
- 每条规则生成验证项和测试场景
- 特殊处理：唯一性校验、联动规则、删除影响等

**输出格式**：
```markdown
#### AC-B1 BR-001: 活动名称在同一分类下唯一

**验证项**:
- [ ] 条件判断: 活动名称在同一分类下唯一
- [ ] 期望行为: 重复时显示错误提示
- [ ] 关联字段 (活动名称, 分类) 联动正确
- [ ] 唯一性校验范围正确

**关联字段**: 活动名称, 分类

**测试场景**:
- **场景 1**: 重复值测试
  - 输入: 输入已存在的值
  - 期望: 显示唯一性校验错误提示
```

### 5. 边界场景验证 (EdgeCaseValidation)

**来源**：`RequirementAnalysis.edgeCases`

**转换规则**：
- 每个边界场景生成验证项
- 特殊处理：空状态、权限不足、超出限制等

**输出格式**：
```markdown
#### AC-E1 活动列表为空

**验证项**:
- [ ] 场景 "活动列表为空" 下，展示: 空状态图标 + "暂无活动" 文案
- [ ] 空状态文案、图标、按钮符合规格
- [ ] 空状态下的操作引导正确

**上下文**: 活动列表页
**兜底行为**: 显示创建活动按钮
```

### 6. UI展示规则验证 (UiDisplayValidation)

**来源**：`RequirementAnalysis.uiDisplayRules`

**转换规则**：
- 按上下文（页面/Tab）分组
- 每条规则生成验证项和视觉检查点

**输出格式**：
```markdown
### 6.1 活动列表页

#### AC-U1.1 列差异: 不同 Tab 显示不同列

**验证项**:
- [ ] 列差异: 不同 Tab 显示不同列: "进行中" Tab 显示开始时间，"已结束" Tab 显示结束时间

**视觉检查点**:
- 列的显示/隐藏符合规格
- 列顺序正确
- 列宽度合理
```

### 7. 功能流程验证 (FunctionalFlowValidation)

**来源**：`RequirementAnalysis.functionalFlows`

**转换规则**：
- 每个流程生成步骤验证项
- 包含条件分支和入口路径验证

**输出格式**：
```markdown
### 7.1 创建活动流程

**完整流程验证**:
- [ ] 步骤 1: 点击"创建活动"按钮
- [ ] 步骤 2: 填写活动信息
- [ ] 步骤 3: 选择分类
- [ ] 步骤 4: 点击"保存"
- [ ] 步骤 5: 显示成功提示并跳转到活动详情页

**条件分支验证**:
- [ ] 保存失败时显示错误提示: 按条件分支执行

**入口路径验证**:
- [ ] 从 活动列表页 触发 → 流程正常启动
- [ ] 从 首页快捷入口 触发 → 流程正常启动
```

## 任务关联映射

### 映射策略

根据任务的以下属性自动匹配验收项：
- `phase`：任务阶段（ui-form → 表单验证、infra → 权限验证等）
- `file`：文件路径（form/modal → 表单验证、components → UI验证等）
- `requirement`：需求描述（关键词匹配）

### 映射规则

| 任务特征 | 匹配的验收项类型 | 匹配条件 |
|----------|-----------------|----------|
| `requirement` 包含 "form/表单/input/输入" | 表单验证 | 字段名或场景名匹配 |
| `phase === 'infra'` 或包含 "权限/permission" | 权限验证 | 角色名或场景名匹配 |
| `phase === 'ui-form'` 或包含 "交互/click/hover" | 交互验证 | 元素名或触发方式匹配 |
| 包含 "规则/rule/校验/validation" | 业务规则验证 | 规则 ID 或描述匹配 |
| `phase === 'ui-display'` 或包含 "展示/display" | UI展示验证 | 上下文或规则匹配 |
| 包含 "边界/异常/error/empty" | 边界场景验证 | 场景描述匹配 |
| 包含 "流程/flow/步骤/创建/编辑" | 功能流程验证 | 流程名称匹配 |

### 输出格式

在 `tasks.md` 中，每个任务包含 `- **验收项**: AC-F1.1, AC-P1.2, ...` 字段，列出关联的验收项 ID。

## 验收通过标准

### 必须满足（Must Pass）

- 所有标记为 "必填" 的字段验证通过
- 所有角色权限验证通过
- 所有业务规则验证通过
- 关键功能流程验证通过

### 建议满足（Should Pass）

- 所有交互行为验证通过
- 所有边界场景验证通过
- 所有UI展示规则验证通过

## 使用场景

### 场景 1：任务执行时参考验收项

执行任务时，开发者可以查看任务关联的验收项，确保实现符合所有验证要求。

```bash
# 查看任务 T3 的验收项
grep "T3:" tasks-xxx.md -A 10 | grep "验收项"
# 输出: - **验收项**: AC-F1.1, AC-F1.2, AC-P1.1

# 查看具体验收项内容
grep "AC-F1.1" .claude/acceptance/xxx-checklist.md -A 10
```

### 场景 2：质量关卡审查

在质量关卡（如 Codex 审查）时，可以参考验证清单进行全面审查。

### 场景 3：验收测试

完成开发后，按照验证清单逐项验收，确保所有需求细节都已实现。

## 注意事项

1. **覆盖率验证**：生成后检查是否所有需求细节都转换为验收项
2. **可执行性检查**：确保每个验收项都有明确的验证标准
3. **关联准确性**：验证任务与验收项的关联是否准确
4. **持续更新**：需求变更时，需要重新生成验证清单
5. **与实现指南同步**：验收清单和实现指南应该保持一致，互相引用
6. **防绕过机制**：执行者应参考以下合理化借口表和红旗清单进行自检

## 合理化借口表

> 常见的跳过验证的借口，以及为什么不能接受。

| 借口 | 为什么不能接受 | 正确做法 |
|------|----------------|----------|
| "手动测试过了" | 手动测试不可重复，无法作为证据 | 运行自动化测试或录制操作步骤 |
| "这个场景不会发生" | 边界场景往往在生产环境才暴露 | 按清单逐项验证，不做假设 |
| "和之前的实现一样" | 上下文不同，行为可能不同 | 在当前上下文中重新验证 |
| "时间不够，先跳过" | 跳过验证的技术债比重写更贵 | 至少完成 Must Pass 项 |
| "测试框架有问题" | 工具问题不是跳过验证的理由 | 修复工具或使用替代验证方式 |
| "改动太小不需要测试" | 小改动也可能引入回归 | 运行相关测试确认无副作用 |

## 红旗清单

> 出现以下信号时，说明验证流程正在被绕过。

- 任务标记为 completed 但没有运行任何验证命令
- 验收项被标记为通过但没有对应的测试输出
- 使用"应该"、"大概"、"看起来"等模糊措辞描述验证结果
- 只验证了正常路径，跳过了所有边界场景
- 引用其他任务的验证结果作为当前任务的证据
- 验证命令的输出没有被读取就声称通过
