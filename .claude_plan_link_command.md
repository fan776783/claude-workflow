# Plan: add `agent-workflow link` for local repo development

## Goal
Add a new CLI command `link` that is explicitly for local development/debugging from the current repository. Per user choice, it should link **all managed directories** (`skills`, `commands`, `prompts`, `utils`, `specs`, `hooks`) from the current repo, while remaining **based on the existing sync architecture** rather than becoming a separate repo-direct install mode.

## Key design decision
Do **not** make agents point directly to the repo in an unmanaged way.
Instead:
1. keep the existing command/install surface and agent-linking helpers,
2. add a dedicated `link` flow that uses the current repo’s `core/` as the source of truth for linking,
3. make `sync` remain the normal “copy to canonical then link agents” mode so it can restore standard behavior.

That means `link` is a deliberate development-mode command, while `sync` remains the stable production/default command.

## Proposed behavior

### `agent-workflow link`
- New CLI command in `bin/agent-workflow.js`.
- Options should mirror `sync` where it matters:
  - `-a, --agent <agents>`
  - `--project`
- Source is always the current repo’s `core/` directory (`path.join(__dirname, '..', 'core')`).
- It should relink the selected agents so that:
  - `commands/agent-workflow/*` point to the repo command entry files
  - each `skills/<skill>` entry points to the repo `core/skills/<skill>` entries
  - internal resources (`prompts/utils/specs/hooks/docs/project`) point into the tool-local `.agent-workflow/` namespace
- It should still preserve current special handling for Claude Code:
  - `CLAUDE.md` copy behavior
  - worktree hook injection for global mode

### `agent-workflow sync`
- Keep existing behavior unchanged as the default path.
- Running `sync` after `link` should restore canonical-backed links by reinstalling canonical and re-linking agents from canonical.

## Implementation strategy

### 1) Extend installer with a repo-link path
Modify `lib/installer.js` by adding a new top-level function, e.g. `linkRepoToAgents(options)`.

This function should:
- resolve target agents the same way `installForAgents()` does
- **skip** `installToCanonical()` entirely
- call shared link helpers using a configurable source root instead of canonical-only assumptions
- write/update `.meta/meta.json` under canonical so status still has a durable metadata anchor

Recommended internal refactor:
- generalize the current linking helpers to work from an arbitrary `sourceRoot`:
  - `linkNonSkillDirs(canonicalDir, agentBaseDir, ...)` → take `sourceRoot`
  - `linkSkillEntries(canonicalDir, agentSkillsDir, ...)` → take `sourceRoot`
  - helper to enumerate skill entries should support `sourceRoot/skills`
- keep `installForAgents()` implemented via the same shared link path, but with `sourceRoot = canonicalDir`
- implement `linkRepoToAgents()` with `sourceRoot = templatesDir`

This minimizes divergence and avoids duplicating the whole linking stack.

### 2) Track source mode in meta
Update meta written in `installForAgents()` / new `linkRepoToAgents()` to record source mode.
Suggested fields:
- `mode: 'canonical-copy' | 'repo-link'`
- `sourceRoot`
- existing `canonicalDir`, `global`, `agents`, `version`, `installedAt`

Why:
- `status` can explain what mode the installation is in
- `doctor` can avoid treating repo-link as broken if canonical content is not populated
- future cleanup logic has an ownership signal instead of inferring from paths only

### 3) Make status aware of repo-link mode
Current `getInstallationStatus()` assumes canonical is authoritative for expected skill names (`lib/installer.js:1126+`).
That will under-report or misclassify repo-link mode if canonical hasn’t been populated recently.

Adjust it to:
- read `.meta/meta.json` first when available
- if `meta.mode === 'repo-link'` and `meta.sourceRoot` exists, derive expected directories from `meta.sourceRoot`
- otherwise keep current canonical-based behavior
- surface a distinct agent mode for display, e.g. `repo-link`

This keeps `status` truthful in both modes.

### 4) Make doctor aware of repo-link mode
Current `doctor` in `bin/agent-workflow.js` hardcodes canonical directory content checks.
For repo-link mode, that would produce false negatives.

Adjust `doctor` to:
- read install status/meta first
- if mode is canonical copy: keep current checks
- if mode is repo-link:
  - verify `meta.sourceRoot` exists
  - verify expected subdirs exist in `sourceRoot`
  - still validate per-agent links using `getInstallationStatus()`
  - message should recommend `agent-workflow sync` to return to normal mode if source repo disappears

## Edge cases to handle

### Stale managed skill cleanup
Current stale cleanup only auto-removes symlinks if target path contains `.agents/agent-workflow/skills/...` (`lib/installer.js:564-601`).
That path test will not catch repo-linked entries.

So for `link` mode, cleanup must not rely solely on canonical-path substring checks.
Recommended change:
- consult manifest ownership first
- or pass the active source root into stale cleanup and treat links pointing inside that root as managed

Without this, renamed/deleted skills in repo-link mode may leave stale agent entries behind.

### Worktree hook injection
`ensureWorktreeHooks()` currently resolves the linked hooks dir and injects `node "~..."` commands.
This should still work if `hooks` points to the repo, but the hook path bug already visible in this session suggests the path handling is brittle.

For this task, do **not** broaden scope into a general hook bug fix unless implementation requires it.
But while wiring `link`, verify that global Claude Code still injects from the effective hooks dir.

### Relative references inside templates
Current templates contain many relative references between `skills/`, `commands/`, `specs/`, `utils/`, and `hooks`, which is why the user requested linking **all managed dirs** rather than only `skills`.
That choice is correct and should be preserved.

### Project vs global mode
- `--project` should link into project-local agent dirs exactly like `sync`
- global mode should behave like current global `sync`
- no interactive flow is required unless you choose to add it later

## Files to change

### `bin/agent-workflow.js`
- add new `link` command beside `sync`
- parse `-a/--agent` and `--project`
- call new installer entrypoint
- print concise result summary similar to `sync`

### `lib/installer.js`
- add repo-link installer entrypoint
- refactor source-root-aware linking helpers
- update stale managed entry cleanup for repo-link ownership
- write meta with mode/source info
- update `getInstallationStatus()` to derive expected entries from active source mode

### `lib/index.js`
- export the new installer API if needed

### `README.md`
- document `link` as a development/debugging command
- clarify that `sync` restores standard canonical-backed installation
- add a short example:
  - `npm run link -- -a claude-code`
  - then `npm run sync -- -a claude-code` to switch back

### `package.json`
- add an npm script only if useful (`"link": "node bin/agent-workflow.js link"`)
- optional, but nice for local development ergonomics

## Suggested user-facing semantics
- `sync` = stable install mode
- `link` = local repo development mode
- if repo path disappears, `status/doctor` should say it is in repo-link mode and recommend `agent-workflow sync`

## Minimal-risk sequencing
1. Refactor link helpers to accept a source root.
2. Add `linkRepoToAgents()`.
3. Add CLI `link` command.
4. Update meta + `getInstallationStatus()`.
5. Update `doctor` output.
6. Update docs/package script.

## Approval notes
This plan intentionally avoids redesigning the whole architecture. It adds a focused development-mode command on top of the existing installer and keeps `sync` as the authoritative way back to normal canonical-backed installs.
