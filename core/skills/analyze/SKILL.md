# 智能技术分析

## 用法

`/analyze <QUESTION_OR_TASK>`

此命令触发技术分析，**不产生代码变更**。

## 角色

**分析协调员**，编排双来源分析：

1. **Auggie** — 代码库上下文检索
2. **Codex** — 外部技术分析候选输入
3. **Claude (Self)** — 一方分析者 + 最终综合裁决者

## 核心原则

- 先基于仓库证据理解问题，再给判断
- `Codex` 输出是**候选分析**，不是最终结论
- 当前模型负责统一裁决：归一化、比对、取舍、综合输出
- 是否后台执行、如何等待结果、是否复用 `sessionId`，统一以 `collaborating-with-codex` skill contract 为准
- Codex 分析契约以 `references/codex-analyzer.md` 为准，不在此重复定义第二套 Prompt 来源
- `/analyze` 调用 Codex 时，**必须**附加 `--read-only` 参数，确保分析过程不产生任何文件写入

## 执行流程

### Phase 1: 上下文检索

1. 调用 `mcp__auggie-mcp__codebase-retrieval` 理解相关代码
2. 识别关键文件、符号、调用路径、架构边界
3. 形成最小上下文包，至少包括：
   - 用户问题
   - 相关文件 / 模块
   - 已识别的关键调用链或契约
   - 当前已知限制或不确定点

若检索结果不足以支撑分析，应先说明证据缺口，而不是直接下强结论。

### Phase 2: 深度判断

根据 Phase 1 的上下文和用户意图，做出两个必选判断和一个可选判断：

1. **`analysis_depth`**（必选）：`locate` | `standard` | `deep`
   - `locate`：定位、解释、简单概览类问题
   - `standard`：需要结构性分析但范围适中的问题
   - `deep`：根因分析、架构评审、性能分析、安全审计等重型分析
2. **`codex_involvement`**（必选）：`skip` | `assist`
   - `skip`：当前模型独立完成（默认姿态，见下方"默认策略与退化"）
   - `assist`：需要 Codex 提供候选技术视角
3. **`focus_hint`**（可选）：一句话描述综合输出应强调什么，仅在 `deep` 深度时有价值

判断依据：以用户问题的语义意图为准。定位类问题默认 `locate`；结构性分析默认 `standard`；根因、架构、性能、安全类问题默认 `deep`。

### Phase 3: 分析执行

根据 Phase 2 的 `codex_involvement` 判断，走不同路径：

#### 当 `codex_involvement: skip` 时

当前模型直接执行独立分析：

- 解释仓库中的真实结构与交互路径
- 分析前端 / UI / UX / CLI / workflow 体验面（若相关）
- 检查跨层契约、调用边界、用户可见影响
- 完成后直接进入 Phase 5（输出），跳过 Phase 4

#### 当 `codex_involvement: assist` 时

并行执行 Codex 候选分析 + 当前模型独立分析：

**Codex 候选分析**：按 `collaborating-with-codex` skill 调用 Codex（**必须附加 `--read-only`**）。
传递内容至少包括：

- 用户问题
- Phase 1 的上下文包
- Phase 2 的判断结果
- 明确的分析范围与非重点范围

Codex 的输出应视为：

- 技术候选观点
- 架构 / 性能 / 安全 / 根因方向的辅助判断
- 后续综合裁决的输入材料

**当前模型独立分析**：在 Codex 分析进行期间立即开始，不等待 Codex 返回。
重点负责：

- 解释仓库中的真实结构与交互路径
- 分析前端 / UI / UX / CLI / workflow 体验面（若相关）
- 检查跨层契约、调用边界、用户可见影响
- 形成独立判断，为后续综合提供对照组

### Phase 4: 裁决与综合（仅 `codex_involvement: assist` 时执行）

在拿到 Codex 候选输出后，当前模型执行统一裁决：

1. **归一化**：将 Codex 与当前模型的观点整理为同一问题空间
2. **共识识别**：找出双方一致支持的发现
3. **分歧裁决**：对结论不一致的点，以仓库证据、调用路径、契约关系为准做取舍
4. **限制说明**：对证据不足或仍有不确定性的判断，显式标注限制条件

若 Codex 输出过于泛化、偏题或缺乏证据支撑：

- 不应直接照搬进最终报告
- 仅保留可验证、可解释的部分
- 其余部分视为未采纳候选输入

### Phase 5: 输出报告

报告结构由 `analysis_depth` 决定，不套固定模板：

#### `locate` — 直接回答

无报告壳。直接给出：

- 定位结果（文件、函数、调用路径）
- 证据链（为什么是这里）
- 若有不确定性，标注

#### `standard` — 轻量报告

推荐章节：

    ## 分析目标
    ## 关键发现
    ## 建议
    ## 限制与不确定性（若有）

#### `deep` — 完整综合报告

推荐章节：

    ## 分析目标
    ## 上下文范围
    ## 候选洞察
    ### Codex（若已调用）
    ### Claude
    ## 共识与分歧（若双方参与）
    ## 风险与限制
    ## 建议

在所有深度下，都可按场景需要补充：性能瓶颈排序、安全风险优先级、待确认问题等。

## 默认策略与退化

**默认姿态：模式 A（当前模型独立完成）。**
只有当以下条件全部满足时，才升级到模式 B：

1. `analysis_depth` 为 `deep`
2. 问题涉及复杂根因、架构边界、后端逻辑、性能、安全、或跨层契约
3. 仅靠仓库检索不足以充分回答

### 模式 A：当前模型独立完成

- 定位类问题
- 小范围结构解释
- 前端 / 交互 / CLI / workflow 分析
- 证据足够且无需额外技术视角的问题

### 模式 B：Codex 协助 + 当前模型综合

- 根因分析
- 架构评审
- 性能分析
- 安全 / 依赖审计
- 跨模块 / 跨层问题
- 调用 Codex 时**必须附加 `--read-only`**

### 模式 C：Degraded 输出

当满足模式 B 的协作条件后，若 Codex 不可用、失败、超时、返回过于泛化，或检索证据不足：

- 区分“已证实发现”与“候选判断”
- 不夸大未证实结论
- 给出最小下一步验证建议

## 注意

- `/analyze` 只做分析，不做代码改动
- 不要在此 skill 中重复定义 `collaborating-with-codex` 的底层执行语义
- 当前模型不是只负责前端；它负责独立分析与最终收口，只是在前端 / 交互相关场景中承担更多直接分析职责
