# Spec: 工作流状态机轻量化重基

> 统一规范文档：需求范围 + 设计决策 + 验收标准

## 1. Context

- 原始需求来源: `/tmp/statemachine-lightweight-prd.md`
- 改造对象: 本仓库自身的 workflow 状态机系统（`core/utils/workflow/*` + `core/hooks/*` + `core/skills/workflow-*`）—— self-hosted 改造。
- 内部依赖: `plan_composer.js`（plan 骨架生成）/ `execution_sequencer.js`（执行引擎）/ `pre-execute-inject.js`（scoped 注入 + dispatch gate）/ `workflow_types.js`（状态/类型）/ `planning_gates.js` + `codex_review_runner.js`（review 触发）/ `doc_contracts.js`（模板 marker validate）。详见 §6 + contract-digest.md。

### 1.1 Problem Statement

4 项目横向对照（superpowers 零状态 / Trellis 轻状态+厚注入 / OMC 多机编排 / 本项目重状态机）证明：**本项目是唯一在 spec-approve 那一刻预渲染带占位 task 的 plan 骨架的**（`cmdSpecReview` → `buildPlanTasks`/`buildTaskBlock` 按"一 requirement 一 task"吐 boilerplate body：假路径 `src/ui/r-xxx.ts`、A1-A3 通用步骤、`npm test` 假命令）。rm-agent 六期实跑证实占位 task body **存活率 0%**——全被 workflow-plan 按 implementation slice 整体重写，却又被 Hard Gate #3 锁死 task ID 造成粒度错配。状态机脊柱本身精简（6 态线性 + halt 环），真正"太强"的是耦合刚性（状态翻转捆绑预渲染骨架 + 锁 `current_tasks[0]` + anchor 编辑，5+ 消费者硬依赖）、gating ceremony、状态/转移冗余（dead `planning` 分支 + halted doc/code drift + review_status 三子审批机）。占位骨架是状态机耦合的副产品，非资产——伤输出质量（模型跟脚手架打架）、费 token（喂占位噪声）。

### 1.2 Assumptions

- 用户已确认走 **B-full 结构性重基**（task-dir 替代 plan.md 作 task 源 + 重写 execution_sequencer），非增量去噪（见 §9.1）。
- 4 项目对照结论与受影响面消费链已在本会话前序逐条核过源码（contract-digest.md 落盘）。
- 改造范围为 CLI/hook/skill 协议层，无前端 UI 交付（`ux_gate_required=false`）。

## 2. Scope

### 2.1 In Scope

- **FR-1 去骨架**：`cmdSpecReview` approve 分支不再调 `buildPlanTasks`/`buildTaskBlock` 预渲染占位 task body；plan.md 不再是结构化 task 来源。`buildTaskBlock` 的 boilerplate（A1-A3 步骤 / `src/ui/r-xxx.ts` 文件桶 / `npm test` 验证命令）整体移除。
- **FR-2 task-dir 作 task 源 + execution_sequencer 重写**：引入 per-task 目录（Trellis 式），承载 task 元数据（id/阶段/package/target_layer/依赖/状态/验收）。`execution_sequencer.js` 的 `loadExecutionContext`/`detectNextTask`/`updateAfterTaskCompletion` 从"`parseTasksV2` 抠 plan.md anchor"重基为读 task-dir。`plan-edit --anchor` / `cmdPlanReview` 的 plan.md-physical 耦合相应迁移。
- **FR-3 JSONL context 背包**：per-task `{file,reason}` 文件指针清单（仅 spec/research 文件，禁 code 路径），复用现有 `contract_digest_path → getContractDigest → <task-contract>` 注入模式，由 `pre-execute-inject.js` 在 scope 解析后展开全文，与 scoped code-specs 并列注入 implementer/check subagent。
- **FR-4 松 Hard Gate #3**：不在 spec-approve 锁死 task ID；task 粒度由 workflow-plan 现写阶段按 implementation slice 定（对齐 4 项目"最后一刻定粒度"共识）。`current_tasks_orphaned_by_edit` guard 随 task 源迁移到 task-dir 语义。
- **FR-5 状态精简 + halted drift 修复**：删 `execution_sequencer.js:122/180` 的 dead `'planning'` 字面量分支（`MINIMUM_STATE_STATUSES` 早已无 planning）；`state-machine.md` 状态表/转换图对齐 code 实际——删 `planning`/`paused`/`blocked`/`failed` 死规范（:14/:17/:18/:19/:36-42），统一为 `halted` + `halt_reason ∈ {failure,dependency,awaiting_codex_review}`。
- **FR-6 机器 review gating 降级为可选**：`codex_spec_review`/`codex_plan_review` 自动触发（`triggerCodexReview`）+ execute 两段 review（spec→code-quality）由默认自动改为**显式开启**（config 开关或命令 flag）。`review_status` 三子审批机相应精简（保留 `user_spec_review` 人工 gate）。
- **FR-7 traceability 降级为 advisory**：Requirement Coverage 表从 `cmdPlanReview` 的硬 `ready` 维度降为 advisory（`coverage.uncovered_ids` 不再卡 ready）；同步删 `doc_contracts.js:11` `REQUIRED_PLAN_TEMPLATE_MARKERS` 中的 `{{requirement_coverage}}` / `## Requirement Coverage` marker（否则 prepublish validate 失败）。
- **FR-8 前置不变量**：补单点守门 `status=planned ⟹ task 源存在`（B-full 下 task 源 = task-dir；替换原"plan_file≠null"假设）。去骨架/惰性化的中间态由该 invariant 统一守门，避免 4 消费者各报不同错。

### 2.2 Out of Scope

- **模型自主度**：不放开 gating 让模型自由决定流程/顺序/何时停。本次只去噪+精简，不改"主会话编排 + subagent 执行"的治理结构。
- **跨平台韧性**：class-2 平台（codex/copilot/gemini/qoder，无 hook）的 pull-based 适配不在本次。
- **4 项目对照分析本身**：是本 spec 的背景论据，非交付物。

### 2.3 Blocked

- 无（Coverage 砍除的唯一硬依赖 `cmdPlanReview.ready` 经核为 skill-driven verdict、不 gate runtime 转移，已解除）。

## 3. Constraints

> 不可协商的硬约束，来自原始需求或技术限制。

- **C-1 resume 不可弃**：`/clear` 后从 disk 重建、跨 session 不丢。resume 锚点 = `current_tasks[0] + status + task 源` 三者，B-full 下"task 源"从 plan.md 迁为 task-dir，三元组语义必须等价保留。这是相对 superpowers 的核心差异价值。
- **C-2 scoped 注入不可弃**：`pre-execute-inject.js` 按 active task 的 `package` + `target_layer` 解析 `getCodeSpecsContextScoped` 注入，直接支撑输出质量。重写 execution 引擎时此注入链必须存活。
- **C-3 invariant 先行**：FR-8 的 `status=planned ⟹ task 源存在` 单点守门必须在去骨架/execution 重写**之前**落地，否则中间态打穿 dispatch gate / sequencer / read-handoff freshness / workflow-status 四消费者。
- **C-4 模板 marker 同步**：动 plan-template / Coverage 必须同步改 `doc_contracts.js` `REQUIRED_PLAN_TEMPLATE_MARKERS`，否则 `scripts/validate.js`（prepublishOnly 唯一门）失败。
- **C-5 对齐 ADR-0002**：execution 重基须与现有 fresh-subagent-per-task 主路径（每 task fresh implementer + 串行 spec→code-quality 两段 review + HITL `NEEDS_CONTEXT` 反问）兼容；FR-6 降级的是"自动触发"，不是删除 review 能力。
- **C-6 保持现有功能不受影响**：spec→plan→execute→delta→archive 全生命周期可用；已 archive 工件不破坏。
- **C-7 无破坏性迁移**：存量 plan.md 格式的旧 workflow（如 8c5fd4f4930b/plans 下历史）需有读取兼容或显式迁移路径，不得静默失效。

> §3.x ADR 注入：**ADR-0002（drop writable parallel → fresh-subagent-per-task）相关**，纳入 C-5；ADR-0001（MCP wrapper 漂移）与本任务无关，仅作历史留痕，可忽略。

## 4. User-facing Behavior

> 用户 = workflow 使用者（开发者）+ 编排主会话模型。"行为"指 spec→plan→execute 各阶段可观测的交互变化。

### 4.1 Primary Flow

改造后的 spec→plan→execute 衔接：

1. `/workflow-spec` 审批通过 → `spec-review --choice approve`：**只**推 `status=planned` + 落 task-dir 结构骨架（task 元数据壳，**无占位 task body**）+ 写 invariant 守护的 task 源。不再渲染 plan.md 占位 task。
2. `/workflow-plan`：现写真实 task 拆分（按 implementation slice 定粒度，不受预锁 task ID 约束），并 curate per-task JSONL 背包（`{file,reason}` 指针）。plan.md 退化为**可选的人类可读叙述**（类 Trellis prd.md），非机器 task 源。
3. `/workflow-execute`：`execution_sequencer` 从 task-dir 读 task 序列；`pre-execute-inject` 按 active task 展开 JSONL 背包全文 + scoped code-specs 注入 implementer subagent。
4. 机器 review（codex / 两段 review）默认**不自动触发**，需显式开启。

### 4.2 Error and Edge Flows

- **invariant 违例**：任何路径产生 `status=planned ∧ task 源缺失` → CLI 统一报 `task_source_missing` 并阻断 execute 入口（替代当前四消费者各报不同错）。
- **存量 plan.md 旧 workflow**：execute 检测到 task 源为 legacy plan.md（无 task-dir）→ 走兼容读取或提示显式迁移（C-7），不静默失效。
- **JSONL 背包指向不存在文件**：`pre-execute-inject` 跳过缺失指针并 stderr warn（对齐 Trellis seed 跳过语义），不阻断注入。

### 4.3 Observable Outcomes

- spec 审批后立即查看 plan：**看不到**机器占位 task 列表（消除"系统没读懂需求"的信任损耗）。
- execute 期 implementer subagent prompt 中 task 上下文来自 JSONL 背包展开的 spec/research 全文，而非 plan.md 占位 body。
- token：plan 阶段不再喂占位 task body / 机械 coverage 表 / 强制 handoff header 噪声。

## 5. Architecture and Module Design

- 以现有 `core/utils/workflow/*` + `core/hooks/*` 为基线，B-full 重基限定在 task 源 + execution 引擎，其余精简为外科式删改。
- 核心架构决策：**plan.md 退化为可选人类可读叙述（类 Trellis prd.md），机器 task 源 = task-dir**。resume 三元组从 `current_tasks[0]+status+plan_file` 等价迁为 `current_tasks[0]+status+task-dir`。

### 5.1 Module Responsibilities

| # | Module | 文件 | 职责 | 改造类型 |
|---|--------|------|------|---------|
| 1 | Plan Composer | `plan_composer.js` | 去 `buildPlanTasks`/`buildTaskBlock` 预渲染；`cmdSpecReview` approve 只推 status + 落 task-dir 壳；保留 front matter/结构锚点最小集 | modify |
| 2 | Task Store | 新增 `task_store.js` | per-task 目录 CRUD：task 元数据 + JSONL 背包指针（Trellis `task.py` 类比） | create |
| 3 | Execution Sequencer | `execution_sequencer.js` | task 源从 `parseTasksV2(plan.md)` 重基为 Task Store 读取；`loadExecutionContext`/`detectNextTask`/`updateAfterTaskCompletion` 改写 | modify |
| 4 | Context Injector | `pre-execute-inject.js` | 保留 scoped code-specs 注入（C-2）；新增 JSONL 背包展开，复用 `getContractDigest` 注入链 | extend |
| 5 | Review Gate | `planning_gates.js` + `codex_review_runner.js` | 自动触发改显式开关；`review_status` 子结构精简（保 `user_spec_review`） | modify |
| 6 | State Types & Invariant | `workflow_types.js` | 删 dead `planning` 分支；加 `status=planned ⟹ task 源` invariant assert；halted 保留 + doc 对齐 | modify |

> module 名取自现有代码 + glossary；新增 Task Store 是唯一新抽象。

### 5.2 Data Models

**task-dir layout**（per-task，落 user 级 `~/.claude/workflows/{pid}/tasks/{taskId}/`）：

| 文件 | 内容 |
|------|------|
| `task.json` | `{id, phase, package, target_layer, depends:[], status, acceptance:[], interaction}` |
| `context.jsonl` | JSONL 背包，每行 `{file: string, reason: string}`（仅 spec/research 路径，禁 code） |

**state schema delta**（`workflow-state.json`）：

- `plan_file` → 语义改为可选叙述文档；新增 `task_source`（指向 task-dir 根）。
- `current_tasks` 保留（执行起点不变）。
- `review_status`：保 `user_spec_review`；`codex_spec_review`/`codex_plan_review`/`plan_review` 仅在显式开启时实例化。
- 新 invariant：`status ∈ {planned,running,halted} ⟹ task_source 存在且非空`。

### 5.3 Technology Choices

- **零新依赖**：沿用 Node CLI + JSON/JSONL 文件状态。
- **JSONL 背包复用** `contract_digest_path → getContractDigest → <task-contract>` 现成注入链，加 per-task `context.jsonl` 展开分支，不另造机制。
- **存储位置不变**：task-dir 落现有 user 级 `~/.claude/workflows/{pid}/`，不改 §存储约定（spec.md 仍可入 git，task 工件仍 user 级）。

### 5.4 Risks and Trade-offs

- **执行引擎重写**（最高）：`parseTasksV2` 是全链路 task 提取根（sequencer/inject/plan-edit/delta 都依赖）。重基需逐消费者迁移，回归面大 → 切片上先建 Task Store + 适配层，再逐个切消费者。
- **resume 三元组等价性**：迁 task 源后 `/clear` 重入必须等价恢复，需专项回归（C-1）。
- **存量 plan.md 迁移**（C-7）：历史 workflow 用 plan.md 作 task 源，需兼容读取层。
- **review 默认关 → silent-done 风险**：FR-6 降级自动 review 后，用 OMC 式"task 完成需双字段判定"补偿（execute 末仍可显式触发终审）。

### 5.5 Depth and Seams

> §5.1 列 6 module（≥3）触发本段。execute 末尾终审（workflow-execute Step 7）的 Depth Heuristics（H1/H2）优先信任本段声明。

#### 5.5.1 Module Depth Justification

每 module 按 H1 deletion test：

| Module | 接口方法数 | Deletion test 结论 |
|--------|-----------|--------------------|
| Task Store | ~5（create/read/list/update-status/curate-context） | 分散到 N 个 caller（删则 sequencer/inject/plan-edit 各自重复读目录 → 真实 deep module，封装 task-dir 布局） |
| Execution Sequencer | 3（loadExecutionContext/detectNextTask/updateAfterTaskCompletion） | 分散到 N 个 caller（执行编排核心，删则散落到 execute skill 各处） |
| Context Injector | hook 入口（buildTaskContext） | 搬到另一处（hook 边界，逻辑可内联但 hook 契约要求独立） |
| Review Gate | ~4（mapSpecReviewChoice/shouldRun*Review/triggerCodexReview） | 搬到另一处（降级后变薄，仍是 review 触发判定的单一收口） |
| State Types & Invariant | 类型 + deriveEffectiveStatus + invariant assert | 分散到 N 个 caller（类型/状态派生/不变量中枢，删则各消费者重复校验） |

> Plan Composer 去 `buildPlanTasks` 后显著变薄，不再列为独立 deep module，归入既有 CLI 命令层。

#### 5.5.2 Seam Strategy

| Seam（接口名） | Adapter 数量 | 理由 |
|---------------|--------------|------|
| `TaskSource`（task 列表来源接口） | 2（`TaskDirSource` 新 + `LegacyPlanMdSource` 存量兼容） | 真实 seam —— C-7 要求存量 plan.md 旧 workflow 不静默失效，故 sequencer 依赖抽象 `TaskSource`，新流程走 task-dir、历史走 plan.md parseTasksV2 兼容读，二者非 single-adapter indirection |

> JSONL 背包不单列 seam：复用现有 `getContractDigest` 注入链，在 Context Injector 内联展开，无新接口。

## 9. Open Questions & Dependencies

### 9.1 需求澄清记录

| 维度 | 问题 | 答案 | 影响 |
|------|------|------|------|
| "太强"指向 | 状态机太强指哪层 | 耦合刚性 + gating 强制度 + 状态/转移本身（grill 戳出脊柱精简，锚定 4 个具体可精简项） | 定义射程边界 |
| 成功标准 | "让模型更好发挥"= 什么 | 输出质量 + 上下文效率（**非**自主度、**非**跨平台） | 排除"放开 gating"路线，定向去噪+精简 |
| 不可弃价值 | 守住哪些状态机价值 | resume 可恢复 + scoped 注入（→ C-1/C-2）；review gating + traceability 可弃 | 限定改造不破 resume/注入 |
| 射程档位 | A 增量 vs B 结构性 | **B 结构性**（向 Trellis-lean 重基） | 选 task-dir + JSONL 路线 |
| B 深度 | B-lite vs B-full vs 最小集 | **B-full 一步到位**（task-dir 替代 plan.md + 重写 execution_sequencer） | 含执行引擎重写，最大改动面 |

### 9.2 方案选择

**选定方案**：B-full 结构性重基 —— task-dir 作机器 task 源 + 重写 `execution_sequencer` 的 `parseTasksV2` 耦合；plan.md 退化为可选人类可读叙述；JSONL 背包接上下文；状态/review/Coverage 精简。

**被排除方案**：

| 方案 | 排除原因 |
|------|---------|
| A 增量（仅 Option B 空骨架 + 折叠冗余态，保状态机形状） | 用户选 B；增量不彻底兑现"质量+效率"，占位骨架根因（plan.md 作 task 源）未除 |
| B-lite（保 plan.md 作 task 源，execution_sequencer 不动） | 用户在 Step 3 明确选 B-full；执行引擎仍耦合 plan.md，去骨架不彻底 |
| 最小集（仅 invariant + 去 boilerplate） | 质量+效率增量最小 |
| C 合并 workflow-spec+workflow-plan skill | 回退 ca77049 审批闸设计意图，且不碰 buildTaskBlock 粒度根因（前序结论） |
