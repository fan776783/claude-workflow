# Stable Skill Contract

适用于所有带阶段门禁、验证门控或状态切换的 skill。

目标只有一个：让 skill 从“说明文档”变成“执行协议”。

## 适用范围

满足以下任一条件时，优先按本约束编写或重构：

- skill 有明确的 Phase / Step 顺序
- skill 要求先产出某些工件，再进入下一步
- skill 包含“通过 / 不通过 / 请求人工判断”这类门控
- skill 经常出现跳步、先实现后补验证、直接宣称完成

## 必备结构

### 1. description 只负责触发

`description` 只写“什么时候该用这个 skill”，不要概述完整流程。

反例：

```yaml
description: Use when implementing Figma UI with asset triage and visual review before delivery
```

这类写法会把完整流程压缩成一句话，模型容易沿着这句捷径执行，而不是继续读正文。

正例：

```yaml
description: Implements UI from a Figma node. Use when given a Figma URL or asked to restore a design draft.
```

### 2. 顶部必须有执行铁律

在正文前部用 3-6 条短规则定义不可绕过的约束。

推荐形式：

- 未完成 Phase A，不得进入 Phase B
- 未拿到验证证据，不得宣称完成
- 出现阻断状态时，必须回退，不得带病继续

如果 skill 的关键约束只有一句话，也要单独成节，不要埋在说明文字里。

### 3. 必须有 Entry Gate

skill 一加载，第一步该做什么，要写成入口门禁，而不是背景说明。

Entry Gate 至少回答三个问题：

- 先确认什么输入
- 先读取什么上下文
- 在入口未完成前，哪些行为禁止发生

### 4. 每个 Phase 都要有 Gate

每个阶段至少写清四件事：

- 输入：本阶段依赖什么
- 必需产物：本阶段结束前必须得到什么 artifact
- 禁止项：本阶段不能做什么
- 放行条件：满足什么条件才允许进入下一阶段

推荐模板：

```markdown
## Phase X: 名称

### 输入
- ...

### 必需产物
- ...

### 禁止项
- ...

### 放行条件
- 同时满足以上产物要求
- 不存在阻断状态
```

### 5. Red Flags 必须显式列出

只写“必须遵守流程”不够。要把常见偷跑想法直接写出来。

常见模板：

- “先做实现，后面再补 artifact”
- “先说完成，回头再跑验证”
- “这次情况特殊，可以跳过”
- “这个 gate 只是建议，不是阻断”

写法要求：出现这些念头时，明确要求回到哪个 phase。

### 6. Exit Criteria 必须基于证据

skill 只要涉及“完成”“通过”“可交付”，就必须定义证据门控。

不要只写：

- “完成后输出摘要”
- “最后检查一下”
- “自行审查后交付”

要写成：

- 需要什么结构化结果
- 哪个分数或状态是通过线
- 低于通过线时如何处理
- 超过多少轮后必须停下并请求用户判断

## 文案禁忌

以下写法很容易让模型忽略后面的约束：

### 禁忌 1：先给放宽语气，再给硬约束

例如：

- “自由编码”
- “不是过程控制者”
- “按需选择流程”
- “可灵活处理”

如果确实存在自由度，只能写在阶段边界内部，不能放在铁律之前。

### 禁忌 2：把可选路径混进主流程

`可选`、`按需`、`降级方案` 不要出现在主链路中部。

更稳的写法是：

1. 先定义默认主路径
2. 再在末尾单列“可选排查 / 降级方案”
3. 明确这些路径不能替代主门禁

### 禁忌 3：用自评代替验证证据

如果 skill 的验证只是“当前模型自行判断差不多”，那它几乎一定会漂。

即使没有外部验证器，也要要求结构化审查结果和明确的过线条件。

### 禁忌 4：阶段产物只列名称，不列放行条件

只写“Phase A 输出：artifact1 / artifact2”还不够。

还要明确：没有这些 artifact，不能进入下一阶段。

## 推荐骨架

```markdown
---
name: xxx
description: Use when ...
---

# Skill Title

一句话说明：默认主路径是什么。

## 执行铁律
- ...

## Entry Gate
- ...

## Phase A
### 输入
### 必需产物
### 禁止项
### 放行条件

## Phase B
### 输入
### 必需产物
### 禁止项
### 放行条件

## Phase C
### 输入
### 必需产物
### 禁止项
### 放行条件

## Red Flags
- ...

## Exit Criteria
- ...

## 可选排查 / 降级方案
- ...
```

## 推广顺序

建议先改最容易跳步、又最依赖阶段门禁的 skill：

1. `figma-ui`
2. `fix-bug` / `debug` 类 skill
3. 各类带 review gate 的 workflow skill
4. 其他说明型但实际承担执行责任的 skill
