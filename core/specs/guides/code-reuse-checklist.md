# 代码复用检查清单

## 触发条件

在以下场景中必须执行此检查：

- 编写与已有代码**功能相似**的新代码
- 修改**常量、配置、枚举**等共享数据
- 创建新的**工具函数或辅助类**
- 添加**错误处理、日志、验证**等横切关注点

## 检查清单

### 1. 是否存在可复用的现有实现？

- [ ] 搜索代码库中是否已有类似功能（用 grep/ripgrep 搜索关键词）
- [ ] 检查 `shared/`、`utils/`、`common/`、`lib/` 等目录
- [ ] 检查项目依赖中是否已引入可用的第三方库
- [ ] 如果找到相似代码，考虑提取为共享模块而非复制

### 2. 修改常量/配置时的影响分析

- [ ] 搜索该常量/配置在代码库中的**所有引用**
- [ ] 确认修改不会破坏其他消费者
- [ ] 如果常量被多处使用，考虑是否需要版本化或渐进式迁移

### 3. 新建工具函数前的必要性检查

- [ ] 确认没有已有的等价实现
- [ ] 确认不能通过组合已有函数实现
- [ ] 确认命名不与已有函数冲突
- [ ] 放置在正确的层级（避免跨层引用）

### 4. 模式一致性

- [ ] 新代码是否遵循项目中已有模式？（错误处理、日志格式、返回值结构）
- [ ] 如果引入新模式，是否有充分理由？是否需要更新 spec？

## 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| 在组件内重写已有的工具函数 | 导入已有模块 |
| 硬编码常量而非使用配置 | 使用统一的常量/配置文件 |
| 复制粘贴相似代码到新文件 | 提取为共享函数并引用 |
| 忽略项目已有的第三方依赖 | 先检查 package.json / requirements.txt |
