  1. 整个工作流编排长什么样

  主流程在 README.md:108 已经写得很清楚，实际仓库实现也基本按这个拆：

  SessionStart 注入 using-superpowers
    -> agent 学会先判断/调用 skill
    -> brainstorming
    -> using-git-worktrees
    -> writing-plans
    -> subagent-driven-development 或 executing-plans
    -> requesting-code-review / TDD / verification 等质量技能穿插
    -> finishing-a-development-branch

  对应证据：

  - 总体流程说明：README.md:108
  - 启动后先学会“怎么用 skills”：skills/using-superpowers/SKILL.md:1
  - brainstorm 完成后只能进入写计划：skills/brainstorming/SKILL.md:66
  - 写完计划后，进入两种执行模式之一：skills/writing-plans/SKILL.md:134
  - 执行完成后进入收尾：skills/executing-plans/SKILL.md:32、skills/subagent-driv
  en-development/SKILL.md:61

  这套编排的关键点

  它不是把“需求分析、设计、计划、执行、审查、收尾”塞进一个 skill。

  而是拆成阶段技能，每个 skill 只负责一个阶段，并且在 skill 文本里写明：
  - 什么时候用
  - 进入条件
  - 退出条件
  - 必须调用的下一个 sub-skill

  这相当于把“流程状态机”写进 skill contract 里，而不是写进代码里。

  ---
  2. superpowers 是怎么拆 skills 的

  我认为它的拆分原则很清晰：按职责边界拆，而不是按工具或平台拆。

  A. 引导/总入口类

  这类 skill 不直接干业务，而是建立流程纪律。

  using-superpowers

  skills/using-superpowers/SKILL.md:1

  作用：
  - 会话一开始加载
  - 强制 agent 在任何回应/行动前先判断是否该调用 skill
  - 明确说了：只要有 1% 可能适用，就必须 invoke skill

  这是整个系统真正的“总入口”。

  ---
  B. 前置设计类

  brainstorming

  skills/brainstorming/SKILL.md:1

  职责：
  - 先澄清需求
  - 提方案对比
  - 分段展示设计给用户确认
  - 产出 spec 文档
  - 审核 spec
  - 然后只允许进入 writing-plans

  这一步本质是把“想做什么”固定下来，禁止直接开写。

  writing-plans

  skills/writing-plans/SKILL.md:1

  职责：
  - 把 spec 拆成可执行计划
  - 每个 task 要求到文件路径、代码、测试命令级别
  - 强调 TDD / DRY / YAGNI / frequent commits
  - 最后把执行权交给：
    - subagent-driven-development
    - 或 executing-plans

  这一步解决的是“怎么做”。

  ---
  C. 执行编排类

  这是 superpowers 最核心的一层。

  subagent-driven-development

  skills/subagent-driven-development/SKILL.md:1

  职责：
  - 读计划
  - 按 task 派发新 subagent
  - 每个 task 后走双重审查：
    a. spec compliance review
    b. code quality review
  - 所有任务完成后再进收尾 skill

  这是它的高自动化主力执行器。

  executing-plans

  skills/executing-plans/SKILL.md:1

  职责：
  - 不走多 subagent 的情况下，按 plan 在当前/独立 session 顺序执行
  - 更偏“串行执行器”

  这两个 skill 是同一执行阶段的两种 runtime：
  - 一个偏 subagent orchestration
  - 一个偏 inline / session execution

  这个拆法很好，因为它把“计划”与“执行策略”分开了。

  ---
  3. 这套 skills 拆分思路为什么有效

  我总结成 4 条：

  1) 一阶段一个 skill

  不是“万能工作流 skill”，而是：
  - 设计一个
  - 计划一个
  - 执行一个
  - 审查一个
  - 收尾一个

  好处是每个 skill 的进入/退出条件很清楚。

  ---
  2) 横切关注点独立成 skill

  例如：
  - TDD
  - code review
  - debugging
  - verification

  这些不是某个阶段独有，所以不应该写死在某个大 skill 里。

  ---
  3) 执行策略与计划内容分离

  writing-plans 负责产出 plan，
  subagent-driven-development / executing-plans 负责消费 plan。

  所以以后要换执行器，不需要重写 planning skill。

  ---
  4) 平台适配不塞进业务 skill

  平台差异放在：
  - plugin manifest
  - hook 配置
  - OpenCode plugin transform
  - 安装文档 / symlink

  业务流程 skill 本身尽量平台无关。

  这点很重要。

  ---
  4. hook 是怎么触发整个流程的

  这里要注意：hook 不是直接执行 brainstorming / writing-plans / 
  subagent-driven-development。

  它做的事情更前置、更关键：

  ▎ 在会话开始时，把 using-superpowers 注入上下文，让 agent 从第一轮开始就按 
  skills 纪律工作。

  ---
  6. 所以 hook 如何“触发流程”

  严格说，它不是直接触发业务流程，而是触发流程治理。

  实际链路是：

  SessionStart
    -> hook 运行 session-start
    -> 注入 using-superpowers 全文
    -> agent 被要求先判断并调用合适 skill
    -> 用户提出任务后，agent 自动进入 brainstorming / debugging / TDD 等
    -> 后续 skill 再显式要求调用下一个 skill

  所以它是：

  hook 负责激活“流程规则”
  skill 负责推进“流程阶段”

  这两个职责分得非常干净。

  9. 我觉得 superpowers 最值得借鉴的编排设计

  1）hook 只做 bootstrap，不做业务决策

  这点很强。

  很多系统会想把 hook 写成“自动判断任务类型然后直接跑流程”，很快就会失控。
  superpowers 没这么做，它只负责把“流程纪律”注入进去。

  ---
  2）真正的状态流转靠 skill contract

  例如：
  - brainstorming 明确说下一步只能是 writing-plans
  - writing-plans 明确交给 subagent-driven-development 或 executing-plans
  - executing-plans / subagent-driven-development 明确结束时必须调用
  finishing-a-development-branch

  这是一种文本化状态机。

  ---
  3）把“流程控制”和“实现内容”分离

  - skill 决定怎么推进
  - plan/spec 决定做什么
  - hook 决定怎么激活

  三层分工非常清楚。

  ---
  4）测试在验证整个编排，而不是只测代码片段

  例如：
  - tests/skill-triggering/run-all.sh:1
  - tests/claude-code/test-subagent-driven-development-integration.sh:13

  它会验证：
  - skill 是否真的被调用
  - 是否真的派了 subagent
  - 是否先做 spec review 再做 code quality review
  - 是否真的创建了产物并跑通测试

  说明这套 workflow 不是只有文档，是真在测“编排行为”。

  ---
  10. 用一句话总结 superpowers 的架构

  superpowers = SessionStart bootstrap + 强约束 skill system + 文本化阶段状态机 
  + 平台适配层。

  如果你重点想复用它的思路，最值得抄的是这三个点：

  1. 用 hook 注入最小但关键的总规则
  2. 把 workflow 拆成阶段 skill，而不是一个大 skill
  3. 让每个 skill 显式声明“下一步必须调用谁”

  这样编排会很稳，而且容易跨平台迁移。

  - 当前禁止的动作是什么

  ---
  一个关键原则：hook 不要拥有“业务判断最终权”

  这点是我最想强调的。

  hook 适合做

  - 注入上下文
  - 校验前置条件
  - 阻断明显违规动作
  - 自动执行机械验证
  - 保护运行时资源

  hook 不适合做

  - 决定需求是否完成
  - 决定 spec 是否合理
  - 决定 plan 怎么拆
  - 决定该走哪个 skill
  - 决定是否进入 team mode / execute mode

  因为这些属于：
  - workflow policy
  - state machine
  - skills contract
  - command semantics

  如果 hook 也管这些，会出现两套真相源。

  推荐方案：workflow orchestrator + hook guardrails

  主控层

  - skill

  - shared runtime spec

  - WorktreeCreate/Remove：并发保护和清理

  这比 superpowers 更适合你们当前项目阶段。

     Hook 驱动 Workflow Guardrails 优化方案

     Context

     当前项目已经具备成熟的 workflow 主体：/workflow command 负责入口路由，专项
     skills 负责阶段协议，workflow-state.json 负责运行时状态机。现有 hooks
     也已覆盖部分运行时守门能力，例如会话启动上下文注入、Task
     派发前上下文补全、质量验证循环、worktree 串行化。

     这次优化的目标不是把主流程改成 hook 驱动，而是在不替换 command / skill / 
     state-machine 架构的前提下，强化 hook 作为 runtime guardrail
     的作用：统一上下文注入、前置拦截、质量卡点与安装注册方式，让 workflow
     的“卡点”更稳定、更可配置、更可测试。

     Recommended Approach

     双重决策。

     关键文件
     - core/specs/workflow-runtime/state-machine.md

     - core/specs/workflow/quality-gate.md

     ree/并发安全。

      私自决定 planning / execute / delta / archive 的阶段流转。

     - 继续以 execute resolver
