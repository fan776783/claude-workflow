### 3.1 代码一致性

**结论**：plan 中 80% 代码是 design 的逐字复制，20% 是结构重组。Plan 相对于 design 几乎没有信息增量。

### 3.3 信息保留率统计

```
Design.md (1018行)
    ↓ 提取代码段 + 重组结构
Plan/1.md (930行)

信息类别          保留率
─────────────────────────
代码块            ~95%
架构决策/推理     ~10%
风险/阻塞项       0%
参考文件列表      0%
实现注意事项      ~30%（散落在各任务"注意事项"中）
待确认项          0%
```

### 3.4 丢失根因

AIPE 的 plan 生成器将 design 当作"代码仓库"提取，只搬运了"做什么"（代码），丢弃了：
- "为什么这样做"（决策推理）
- "可能出什么问题"（风险预案）
- "去哪找参考"（文件索引）
- "什么还没确认"（阻塞项）

这与信息的实际价值**成反比**——代码最容易重新生成，决策和风险最难重新推导。

## 4. 两种 Plan 的结构对比

### 4.1 信息密度

| 维度 | Agent-Workflow plan | AIPE plan |
|------|--------------------:|----------:|
| 单任务平均长度 | ~30 行 | ~120 行 |
| 代码占比 | 0% | ~70% |
| 执行者自由度 | 高（描述 what，how 由执行者决定） | 低（直接给代码，执行者 copy-paste） |
| 变更传播成本 | 改一行步骤描述 | 改 N 处代码段 + 关联注释/样式 |

### 4.2 质量保障机制

| 机制 | Agent-Workflow | AIPE |
|------|---------------|------|
| 需求覆盖矩阵 | R-001~R-007 × Task 映射表 | 无 |
| 验证命令 | 每 task 有 grep/ls/pnpm 命令 + 期望输出 | 无 |
| 依赖声明 | 显式 T1→T2→T3 | 隐含在任务顺序中 |
| 质量关卡 | 布尔标记（true/false） | 无 |
| 人工介入点 | HITL 标注（T3, T7 需手动验证） | 无 |
| Spec 回溯 | 每 task 标注 `Spec 参考: §x.x` | 无 |

### 4.3 Spec-Plan 互补 vs. Design-Plan 冗余

**Agent-Workflow 模型**：
```
Spec（为什么 + 验收标准）
  ↓ 互补
Plan（怎么做 + 验证命令）
  → 执行器读 plan 步骤 + 回查 spec 约束
```

**AIPE 模型**：
```
Design（为什么 + 怎么做 + 完整代码）
  ↓ 有损复制
Plan（怎么做 + 完整代码）
  → 执行器读 plan = 读 design 的子集
```

Agent-Workflow 的 spec 和 plan 是**正交互补**关系；AIPE 的 design 和 plan 是**包含冗余**关系。

## 5. AIPE "参考代码"的多余性分析

### 5.1 为什么参考代码是过度设计

| 论点 | 说明 |
|------|------|
| AI 执行器不需要参考代码 | 给约束和上下文即可，预写代码反而限制执行器适应实际代码结构（如 useTemplateForm 签名变化） |
| 人类执行器也不需要此粒度 | Design 应是架构图 + 接口契约 + 关键决策，完整 SFC 是实现产物不是设计产物 |
| 信息过期速度快 | 项目每天在变，引用的组件名/props/store 方法随时可能改 |
| 变更成本高 | 需求变 → design 改代码 → plan 改代码 → 执行改代码，三层冗余传播 |
| 假精确 | 代码引用具体 props/方法签名但未验证当前是否存在/匹配 |
| 决策未锁定就写代码 | scene=12 标注"待确认"但代码全部硬编码 12 |

### 5.2 适用场景判断

| 场景 | 推荐方案 |
|------|---------|
| AI agent 自主执行 | Agent-Workflow（约束驱动，执行器自适应） |
| 人类开发者手动实现 | AIPE design 的**关键片段**有参考价值，但应收敛为接口签名级别 |
| 需求频繁变动 | Agent-Workflow（改步骤描述成本低） |
| 需求稳定 + 新人执行 | AIPE plan 作为"教程"有一定价值 |

## 6. 改进建议

### 6.1 对 AIPE 的建议

1. **Design 层去代码化**：design 只保留架构图、接口契约、数据模型、关键决策和风险项；代码由执行阶段按需生成
2. **Plan 层增加元信息**：依赖声明、验证命令、阻塞项标注、需求 ID 回溯
3. **锁定决策再写代码**：待确认项未解决时不应生成硬编码的实现代码
4. **Design → Plan 保留决策链**：plan 应继承 design 的风险项和注意事项，而非只继承代码
5. **引入参考文件索引**：plan 每个任务应标注 `Patterns to Mirror` 和 `Mandatory Reading`

### 6.2 对 Agent-Workflow 的借鉴

AIPE design 中以下能力值得 agent-workflow 参考：
- **类图/Mermaid 可视化**：组件关系一目了然
- **依赖关系表**：外部包 + 内部模块的完整映射
- **优化建议章节**：记录"不做但值得未来做"的方向

## 7. 总结

| 维度 | Agent-Workflow 优势 | AIPE 优势 |
|------|-------------------|-----------|
| 执行效率 | 约束驱动，AI 自适应 | — |
| 变更韧性 | 改步骤描述成本低 | — |
| 质量保障 | 验证命令 + 覆盖矩阵 + HITL | — |
| 决策追溯 | spec-plan 正交互补 | — |
| 代码参考 | — | 新人可直接对照实现 |
| 可视化 | — | 类图 + 依赖表 |
| 全景视图 | — | 风险分析 + 优化建议（虽 plan 丢失） |

**核心判断**：对于 AI agent 执行的工作流，AIPE 的"提前生成参考代码"模式是一个**反模式**——它用最容易重新生成的产物（代码）占据了制品的主要篇幅，却在传递中丢弃了最难重新推导的产物（决策、风险、阻塞项）。Agent-Workflow 的「约束 + 步骤 + 验证」模式在执行效率、变更韧性和质量保障三个维度均优于「参考代码搬运」模式。
