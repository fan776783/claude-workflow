# 先搜后写

## 用法

`/search-first <功能需求描述>`

在动手实现前，系统性搜索现有解决方案。

## 核心原则

- 不重复造轮子
- 搜索范围：代码库内 → npm/PyPI → GitHub → MCP 工具
- 产出明确的 Adopt / Extend / Build 决策

## 执行流程

### Step 1: 需求分析

1. 识别核心能力需求（输入/输出/约束）
2. 提取搜索关键词（功能、技术术语、场景）
3. 确定搜索范围

### Step 2: 并行搜索

以下搜索可并行执行：

| 来源       | 方法                                  | 重点                 |
| ---------- | ------------------------------------- | -------------------- |
| 代码库     | `mcp__auggie-mcp__codebase-retrieval` | 已有实现、可复用组件 |
| npm / PyPI | Web 搜索                              | 成熟的开源包         |
| GitHub     | Web 搜索                              | 参考实现、最佳实践   |
| MCP 工具   | 检查已安装 MCP server                 | 可直接调用的工具     |

### Step 3: 评估

对每个候选方案评估：

| 维度     | 问题                 |
| -------- | -------------------- |
| 适配度   | 是否满足核心需求？   |
| 维护性   | 最近更新？活跃社区？ |
| 依赖代价 | 引入多少新依赖？     |
| 集成成本 | 接入需要多少改造？   |
| 许可证   | 是否与项目兼容？     |

### Step 4: 决策

| 决策       | 条件                        | 输出                |
| ---------- | --------------------------- | ------------------- |
| **Adopt**  | 现有方案满足 >80% 需求      | 安装命令 + 集成示例 |
| **Extend** | 现有方案满足 50-80%，可扩展 | 基础包 + 扩展方案   |
| **Build**  | 无合适方案，或集成代价过高  | 自建方案 + 参考代码 |

## 与 workflow-planning 的集成

`/workflow plan` 的 Phase 0 代码分析阶段可主动触发 search-first，特别是在识别到以下需求时：

- 外部 API 集成
- 数据处理/转换
- 认证/授权
- 文件格式解析
