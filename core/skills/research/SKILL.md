# Research

不重复造轮子 + 不基于过期知识做决策。

## 用法

```bash
/research <需求 / 研究主题>           # 自动路由
/research --internal <功能需求>        # 仅代码库 + npm/PyPI
/research --external <研究主题>        # 仅外部引文
```

## 自动路由

| 信号 | 模式 |
|------|------|
| "有没有现成 / 工具 / 组件 / MCP / 库"、具体功能描述 | internal |
| "最佳实践 / 技术选型 / 对比 / trade-off"、概念研究 | external |
| 既要找库又要对比方案 | 混合(先 internal 再 external) |

不确定时先问用户一句。

## 规则(两种模式共用)

- 每个断言附来源(内部 = 文件 + 行号;外部 = URL + 日期)
- 优先近期来源(外部 1 年内;内部看 git log 活跃度)
- 交叉验证关键信息,不确定的明确标为不确定
- 并行搜——不要顺序等一个源搜完再搜下一个

## Internal 模式

**搜索范围**:

| 来源 | 方法 |
|------|------|
| 代码库 | 本地代码检索 |
| npm / PyPI | Web 搜索 |
| GitHub | Web 搜索 |
| MCP 工具 | 检查已安装 MCP server |

**评估维度**: 适配度 / 维护性(最近更新?) / 依赖代价 / 集成成本 / 许可证

**决策**:

| 结论 | 条件 | 输出 |
|------|------|------|
| **Adopt** | 满足 > 80% | 安装命令 + 集成示例 |
| **Extend** | 满足 50–80%,可扩展 | 基础包 + 扩展方案 |
| **Build** | 无合适或集成代价过高 | 自建方案 + 参考代码 |

## External 模式

**工具优先级**: `mcp__firecrawl__*` → `mcp__exa__search` → `read_url_content` → Web 搜索。不可用时自动降级。

**产出格式**:

```markdown
# 研究报告: [主题]

## 摘要
[2-3 句核心发现]

## 关键发现

### [发现 1]
[内容] [^1]

### [发现 2]
[内容] [^2]

## 建议
[基于研究的行动建议]

## 不确定性与限制
[未证实的信息]

## 参考来源
[^1]: [URL] - [日期] - [标题]
[^2]: [URL] - [日期] - [标题]
```

**质量底线**: 核心断言至少 1 个来源;官方文档优先;过期信息(> 2 年)标注;矛盾信息列多方观点。

## 与其他 skill 的关系

- `/workflow-spec` 代码分析阶段可触发本 skill
- `/grill` 质询中需要外部证据 → 跳到本 skill
- `/fix-bug` 怀疑第三方库 bug → `--external` 查 issue / changelog
