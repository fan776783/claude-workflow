# Out-of-Scope Protocol

记录被明确拒绝的需求/建议及拒绝原因。类似需求再次出现时，skill 检查此目录避免重复讨论。

## 目录位置

项目根目录 `.out-of-scope/`。每个被拒需求一个 `.md` 文件。

## 文件格式

```markdown
# {被拒需求的简短标题}

**拒绝日期**: YYYY-MM-DD
**来源**: <issue / 对话 / PR URL>

## 需求描述

<一句话说想做什么>

## 拒绝原因

<为什么不做——不是"暂时不做"而是方向不对>

## 重新评估条件

<什么变化会让这个需求重新值得考虑，或 "None — 永久拒绝">
```

## 哪些 skill 检查

- `/workflow-spec` — Step 2 代码库分析末尾
- `/improve-architecture` — Explore 阶段开始前

命中时告知用户曾被拒绝及原因，由用户决定是否重新评估。不命中则跳过，不提。

## 写入时机

- `/workflow-spec` 需求被用户否决
- Code review 中建议被否决且原因是方向性的

## 不写

- "现在不做"（时机问题）
- "太复杂"（精力问题）
- 没有明确原因的拒绝
