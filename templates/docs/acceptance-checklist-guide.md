# 验证清单使用指南

## 快速开始

### 1. 生成验证清单

验证清单在 `/workflow start` 时自动生成（如果需求文档长度 > 500 字符）：

```bash
/workflow start docs/prd.md
```

生成的文件：
- `.claude/acceptance/{name}-checklist.md` - 验证清单
- `.claude/tech-design/{name}.md` - 技术方案
- `~/.claude/workflows/{projectId}/tasks-{name}.md` - 任务清单

### 2. 查看任务关联的验收项

在任务清单中，每个任务都包含 `验收项` 字段：

```markdown
## T3: 实现活动创建表单
- **阶段**: ui-form
- **文件**: `src/components/ActivityForm.vue`
- **需求**: 实现活动创建表单，包含名称、分类、时间等字段
- **验收项**: AC-F1.1, AC-F1.2, AC-F1.3, AC-B1
- **actions**: `create_file`
- **状态**: pending
```

### 3. 查看具体验收项内容

打开验证清单文件，搜索验收项 ID：

```bash
# 使用 grep 快速查找
grep "AC-F1.1" .claude/acceptance/activity-management-checklist.md -A 15

# 或直接打开文件查看
code .claude/acceptance/activity-management-checklist.md
```

### 4. 执行任务时参考验收项

在实现任务时，参考关联的验收项确保：
- 所有必填字段都有验证
- 所有校验规则都已实现
- 所有提示文案都符合规格
- 所有边界场景都有处理

### 5. 验收测试

完成任务后，按照验证清单逐项验收：

```markdown
#### AC-F1.1 活动名称

**验证项**:
- [x] 活动名称 为空时，显示提示: "活动名称为必填项"  ✅ 已验证
- [x] 活动名称 字符限制 50 字符  ✅ 已验证
- [x] 输入框显示 placeholder: "请输入活动名称"  ✅ 已验证

**测试数据**:
| 输入 | 期望结果 | 实际结果 | 状态 |
|------|----------|----------|------|
| （空值） | 显示错误提示 | ✅ 符合预期 | Pass |
| 超过 50 字符 | 禁止输入 | ✅ 符合预期 | Pass |
```

## 常见场景

### 场景 1：表单字段实现

**任务**：实现用户注册表单

**关联验收项**：
- AC-F1.1 用户名（必填、长度限制、唯一性）
- AC-F1.2 邮箱（必填、格式校验）
- AC-F1.3 密码（必填、强度校验、确认密码）

**实现检查清单**：
1. ✅ 所有必填字段都有 `required` 属性
2. ✅ 所有字段都有 `maxLength` 限制
3. ✅ 所有字段都有对应的 `validationMessage`
4. ✅ 所有字段都有 `placeholder` 提示
5. ✅ 提交时触发完整的表单校验
6. ✅ 校验失败时显示明确的错误提示

### 场景 2：权限控制实现

**任务**：实现角色权限控制

**关联验收项**：
- AC-P1.1 管理员 - 删除用户
- AC-P1.2 管理员 - 编辑用户
- AC-P2.1 普通用户 - 查看用户列表（仅自己）

**实现检查清单**：
1. ✅ 按钮/入口根据角色显示/隐藏
2. ✅ API 层面有权限校验
3. ✅ 数据范围过滤正确（仅显示有权限的数据）
4. ✅ 无权限时显示明确的提示信息
5. ✅ 权限不足时不能绕过前端限制

### 场景 3：交互行为实现

**任务**：实现删除确认交互

**关联验收项**：
- AC-I1.1 删除按钮 - click
- AC-I1.2 确认弹窗 - 显示/关闭

**实现检查清单**：
1. ✅ 点击删除按钮显示确认弹窗
2. ✅ 弹窗内容清晰明确
3. ✅ 确认后执行删除操作
4. ✅ 取消后关闭弹窗，不执行删除
5. ✅ 删除成功后显示成功提示
6. ✅ 删除失败后显示错误提示

### 场景 4：业务规则实现

**任务**：实现活动名称唯一性校验

**关联验收项**：
- AC-B1 活动名称在同一分类下唯一

**实现检查清单**：
1. ✅ 输入时/失焦时触发唯一性校验
2. ✅ 校验范围正确（同一分类下）
3. ✅ 重复时显示明确的错误提示
4. ✅ 提示信息包含冲突的活动名称（如适用）
5. ✅ 编辑时排除自身（不与自己冲突）

### 场景 5：边界场景处理

**任务**：实现空状态展示

**关联验收项**：
- AC-E1 活动列表为空
- AC-E2 搜索无结果

**实现检查清单**：
1. ✅ 空状态图标正确显示
2. ✅ 空状态文案清晰明确
3. ✅ 提供操作引导（如"创建活动"按钮）
4. ✅ 不同场景的空状态有区分（列表为空 vs 搜索无结果）

**关联验收项**：

**实现检查清单**：
1. ✅ 列的显示/隐藏符合规格
2. ✅ 列顺序正确
3. ✅ 列宽度合理
4. ✅ 固定列在滚动时保持固定
5. ✅ 响应式布局正确

**实现检查清单**：

## 最佳实践

### 1. 实现前先看验收项

在开始实现任务前，先查看关联的验收项，确保理解所有验证要求。

### 2. 边实现边验证

不要等到全部实现完成后再验证，而是边实现边验证，及时发现问题。

### 3. 使用测试数据

验证清单中提供了测试数据，直接使用这些数据进行测试，确保覆盖所有场景。

### 4. 记录验收结果

在验证清单中记录验收结果，包括验收人、验收时间、验收状态等。

### 5. 发现问题及时反馈

如果发现验证清单有遗漏或错误，及时反馈并更新。

## 常见问题

### Q1: 验证清单太长，如何快速定位？

**A**: 使用 grep 或编辑器的搜索功能，搜索验收项 ID（如 AC-F1.1）或关键词。

### Q2: 任务没有关联验收项怎么办？

**A**: 如果任务没有关联验收项，说明该任务可能是基础设施类任务（如配置、工具函数等），可以参考通用验收标准：功能正常、无报错、符合设计规格。

### Q3: 验收项与实际需求不符怎么办？

**A**: 验证清单是基于需求文档自动生成的，如果与实际需求不符，可能是需求文档不够详细或有歧义。建议：
1. 与产品确认实际需求
2. 更新需求文档
3. 重新生成验证清单

### Q4: 如何更新验证清单？

**A**: 如果需求变更，需要重新执行 `/workflow start` 生成新的验证清单。建议使用 `/workflow delta` 进行增量变更。

### Q5: 验证清单可以手动编辑吗？

**A**: 可以手动编辑，但建议只做小的调整。大的变更建议更新需求文档后重新生成。
